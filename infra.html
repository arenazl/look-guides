Deploy continuo — cómo quedó el flujo

Heroku

Auto-deploy nativo de GitHub. Un git push a la rama publica solo. Cero YAML, gratis.

Netlify (fronts)

Build-on-push: cada push reconstruye y publica el frontend. 20+ sitios conectados a su repo.

GCP Cloud Run

Cloud Build trigger por servicio: push a main → build + deploy automático. Sin scripts en el repo.

Detalle técnico: los triggers de Cloud Build 2.ª gen exigen un --service-account explícito (cambio de Google 2024) y la conexión a GitHub requiere un token OAuth por proyecto (se reusó el de un proyecto piloto de forma no interactiva). Los backends Python sin Dockerfile fallaban porque el builder usa Python 3.13/3.14; se resolvió fijando la versión con un Dockerfile (python:3.11-slim).

API Keys (Gemini) — auditoría

Riesgo: key "fantasma"

Una misma key la comparten 6 apps y no vive en ningún proyecto GCP del usuario. Punto único de falla: si se cae, caen las 6.

Keys sin uso

3 keys registradas que no las usa nadie (dbsync-prod, salesbot-prod, sinvueltasya-prod). Borrables.

Bien guardadas

Las apps de Cloud Run usan Secret Manager (seguro). Las de Heroku, texto plano en config (mejorable).

El método: se cruzó el valor real de cada key (no por nombre) entre lo registrado en GCP y lo que usa cada backend. Hallazgo clave: el nombre de una key no indica qué app la usa (la key llamada "ngcheck" o "pagealert" en realidad no las usa nadie). Detalle completo en INFORME_KEYS.md.

Lo que queda de gasto es básicamente la IA (Gemini, ~US$29) — eso es valor, no desperdicio. Detalle en REPORTE_COSTOS.html.

Por qué GCP costaba más que Heroku: en Heroku las apps estaban dormidas (gratis); en GCP están despiertas con tráfico real, se sumó la IA, y había servicios fijados a 24/7. Aprendizaje: las cargas 24/7 constantes son más baratas en un dyno fijo de Heroku; las esporádicas, en Cloud Run (escala a cero).
